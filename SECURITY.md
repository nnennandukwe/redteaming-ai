# Security Policy

## Scope

This repository includes intentionally vulnerable demo code for educational and
testing purposes. That does not mean all security problems should be reported
publicly. Issues that could create a real exploit path, expose credentials, or
affect users running the project outside the documented demo context should be
reported privately.

## Supported Versions

Security fixes are handled on a best-effort basis for the latest state of the
default branch. Older commits and unmaintained forks may not receive fixes.

## Reporting a Vulnerability

- Do not open a public GitHub issue for sensitive vulnerabilities.
- Use GitHub's private security reporting flow for this repository when
  available.
- If private reporting is not available, contact the maintainers privately
  through GitHub and include enough detail to reproduce the issue.

When reporting a vulnerability, include:

- A clear description of the issue
- Impact and affected components
- Reproduction steps or proof of concept
- Suggested mitigations, if known

## What to Expect

- Reports will be reviewed as quickly as possible.
- Maintainers may ask follow-up questions or request a private reproducer.
- Please avoid public disclosure until a fix or mitigation is available.

## Demo-Specific Note

If the behavior you found is an intentional vulnerability already documented as
part of the demo, it may be treated as product behavior rather than a security
defect. Please still report it privately if you believe it creates an
unexpected real-world risk.
